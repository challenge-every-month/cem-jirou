# データベース接続・バージョンAPI 要件定義書

## 概要
Cloudflare D1 (SQLite) を使用したデータ永続化基盤を構築し、システムの稼働状態およびバージョン情報を取得・更新するための API を提供する。

## ユーザストーリー

### ストーリー1: システム稼働の確認
- **である** 開発者・システム **として**
- **私は** API を通じて DB のバージョン情報を取得したい
- **そうすることで** アプリケーションと DB が正常に接続されていることを保証できる

### ストーリー2: バージョン情報の更新（デバッグ用）
- **である** 管理者 **として**
- **私は** API を通じて DB 内のバージョン情報を更新したい
- **そうすることで** DB 移行やメンテナンスのステータスを手動で制御できる

## 機能要件（EARS記法）

### 通常要件
- REQ-DB-001: システムは Cloudflare D1 をデータストレージとして使用しなければならない
- REQ-DB-002: システムは `GET /version` リクエストに対して、DB から取得したバージョン情報を JSON 形式で返さなければならない
- REQ-DB-003: システムは `POST /version` リクエストに対して、指定された新しいバージョン番号で DB のデータを更新しなければならない

### 条件付き要件
- REQ-DB-101: DB 接続が失敗した場合、システムは `500 Internal Server Error` とエラーメッセージを返さなければならない
- REQ-DB-102: 未定義のルートへのアクセスがあった場合、システムは `404 Not Found` を返さなければならない

### 制約要件
- REQ-DB-401: データベース操作は D1 の SDK/API を通じて非同期で行わなければならない
- REQ-DB-402: テーブルスキーマは外部ファイル (`schema.sql`) で定義され、Wrangler を通じてマイグレーション可能でなければならない

## 非機能要件

### パフォーマンス
- NFR-DB-001: DB クエリを含む API レスポンスは 200ms 以内に完了しなければならない（コールドスタート時を除く）

### 保守性
- NFR-DB-201: D1 のバインディング名は一貫して `DB` としなければならない

## Edgeケース
- EDGE-DB-001: バージョン情報のテーブルが存在しない場合に、適切な初期化またはエラーハンドリングを行う
- EDGE-DB-002: 巨大な文字列がバージョンとして送信された場合、サイズ制限（例: 255文字）を適用する

## 受け入れ基準
- [ ] `npx wrangler d1 execute` でローカルおよび本番の DB にテーブルが作成できること
- [ ] `GET /version` で初期値（例: v0.0.1）が取得できること
- [ ] `POST /version` で値が変更され、その後の GET で反映を確認できること
